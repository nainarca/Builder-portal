| ADR-013 | Data Access Abstraction: Repository Pattern as the mandatory, sole interface to Supabase | **Accepted** | NG-007, 2026-07-09 — no Feature, Service, or Component ever calls the Supabase client directly; only Shared-tagged Repositories do, one per in-scope Information Domain, mechanically enforced by the Nx tag matrix (NG-002/NG-003). Excludes Properties/Loans/Expenses (A-007 ID-13) entirely, by the same Restricted-Financial rule as every prior document. | ADR-014 | Zoneless Change Detection: `zone.js` removed platform-wide | **Accepted** | NG-009, 2026-07-09 — enabled directly by ADR-001 (Standalone Components) and ADR-004 (Signals) both already being decided: every state category already has exactly one Signal owner that notifies Angular's renderer directly, making `zone.js`'s automatic async dirty-checking redundant. Shrinks the initial bundle and removes a class of unnecessary change-detection cycles. See `NG-009_Performance_Scalability_Architecture.md` §14, `PERFORMANCE_STRATEGY.md` §17 |
| ADR-015 | Hybrid Rendering by Application: SSR + prerendering for Public Website, CSR-only for Super Admin and Builder Portal | **Accepted** | NG-009, 2026-07-09 — decided per-application because the deciding factor (SEO relevance, unauthenticated-visitor latency sensitivity) is true of exactly one of the three applications. See `NG-009_Performance_Scalability_Architecture.md` §14, `PERFORMANCE_STRATEGY.md` §19 |
| ADR-016 | Centralized Error Handling: Global Error Handler + one HTTP/Supabase interceptor as the only two error catch points platform-wide | **Accepted** | NG-010, 2026-07-09 — formalizes `TECHNICAL_STANDARDS.md` §11's principle (no local `try/catch` swallowing) into a mandatory, ADR-backed architecture, the same "principle stated early, formalized later" pattern NG-007 used for the Repository Pattern (ADR-013). | ADR-017 | Immutable Build Artifact Strategy: Build Once, Promote Many | **Accepted** | NG-011, 2026-07-09 — one build artifact per application, per release candidate, built exactly once and promoted unmodified through QA, UAT, Staging, and Production. ## Open items carried from prior documents that may become future ADRs

- Whether the Organization data model needs an explicit `organization_type` discriminator (A-002 §16, A-003 §16) — likely an ADR once the Builder Handover Portal's data architecture document is written, not before.
- **ADR-008 (candidate) — Support Access Invocation Mechanism** (A-008 §7): the *policy* is fully placed within the identity model (NG-006's ADR-012 — an extension of Super Admin Context). NG-007 `RPC_STRATEGY.md` §6 has now named the invocation as the natural RPC candidate (a multi-step, atomicity-sensitive operation — grant record, audit event, expiry — belonging behind one RPC call), but the actual invocation *workflow* (who approves it, duration limits, request mechanism, the RPC's own design) remains undesigned — NG-007 explicitly named it without designing it, consistent with its own no-SQL/no-implementation scope. **Reconfirmed still without an owning trigger document by NG-014's terminal review** (`TECHNICAL_DEBT_REGISTER.md` G-05) — passed over by every candidate document through NG-013; NG-014 deliberately did not design it either, consistent with its own audit-only scope, and instead names it as a Medium-severity, narrowly-scoped implementation blocker requiring an explicit owner. **Renumbered from an earlier provisional "ADR-006" reference in A-008 to ADR-008 during NG-000, to avoid a collision — not left standing.**
- **No prior document specifies how the platform's first Super Admin account is created** (surfaced by NG-006, `AUTHENTICATION_ARCHITECTURE.md` §3) — a genuine operational gap, not yet assigned an owner or a trigger document. **Reconfirmed still open by NG-014's terminal review** (`TECHNICAL_DEBT_REGISTER.md` G-06).
